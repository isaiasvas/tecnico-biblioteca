O que é HTTP?

HTTP (HyperText Transfer Protocol) é o protocolo de comunicação entre clientes e servidores na web. É como um "idioma" que seu browser e um servidor Django falam juntos.

Cliente
Faz a requisição
Servidor
Processa e responde
Protocolo
Regras da conversa
Stateless
Cada request é independente

Anatomia de uma requisição HTTP

GET /api/produtos/1/ HTTP/1.1
Host: meusite.com
Authorization: Bearer eyJhbGci...
Content-Type: application/json
Accept: application/json

Anatomia de uma resposta HTTP

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "nome": "Notebook Pro",
  "preco": 2999.90
}
💡 No Django REST Framework, você não precisa escrever essas strings manualmente — o framework monta e interpreta isso por você automaticamente.

O que é CRUD?

CRUD descreve as 4 operações básicas de qualquer sistema que trabalha com dados. Toda API que você vai criar no Django vai girar em torno dessas 4 ações.

C — Create
Criar registro

Adicionar novo produto, usuário, pedido...

R — Read
Ler / listar

Buscar um produto, listar todos...

U — Update
Atualizar registro

Editar preço, nome, status...

D — Delete
Deletar registro

Remover produto, conta, item...

CRUD mapeado para HTTP + Django

CRUD Método HTTP URL típica View Django
CreatePOST/api/produtos/Recebe JSON, cria objeto no banco
Read (lista)GET/api/produtos/Retorna lista de objetos
Read (detalhe)GET/api/produtos/1/Retorna um objeto específico
Update (total)PUT/api/produtos/1/Substitui todos os campos
Update (parcial)PATCH/api/produtos/1/Atualiza só os campos enviados
DeleteDELETE/api/produtos/1/Remove o objeto do banco
💡 Percebem que a URL /api/produtos/ e /api/produtos/1/ repetem? Só o método HTTP muda o comportamento — isso é o coração do REST.

REST vs RESTful — qual a diferença?

REST (Representational State Transfer) é um conjunto de princípios arquiteturais definidos por Roy Fielding em 2000. RESTful é como chamamos uma API que segue esses princípios corretamente.

❌ Não RESTful
GET /pegarTodosProdutos
POST /criarNovoProduto
GET /deletarProduto?id=1
POST /atualizarProduto/1
✅ RESTful
GET /api/produtos/
POST /api/produtos/
DELETE /api/produtos/1/
PUT /api/produtos/1/

Os 6 princípios REST

# Princípio O que significa
1StatelessCada requisição carrega toda informação necessária — o servidor não guarda sessão
2Client-ServerFrontend e backend são separados e se comunicam apenas via API
3Uniform InterfaceURLs representam recursos; métodos HTTP definem a ação
4CacheableRespostas podem ser cacheadas para melhorar performance
5Layered SystemO cliente não sabe se fala com o servidor final ou um intermediário
6Code on Demand(Opcional) Servidor pode enviar código executável ao cliente

Recursos vs Ações — a virada de chave

No REST, a URL identifica um recurso (substantivo), e o método HTTP indica a ação (verbo).

# ✅ REST — substantivos nas URLs
GET    /api/produtos/        # → listar produtos
POST   /api/produtos/        # → criar produto
GET    /api/produtos/1/      # → detalhe do produto 1
PUT    /api/produtos/1/      # → substituir produto 1
PATCH  /api/produtos/1/      # → atualizar parcialmente
DELETE /api/produtos/1/      # → deletar produto 1

Os 5 métodos HTTP no Django REST Framework

Método Idempotente? Tem body? Uso no Django Retorno típico
GET✅ Sim❌ NãoListar ou detalhar objetos200 OK
POST❌ Não✅ SimCriar novo objeto201 Created
PUT✅ Sim✅ SimSubstituir objeto inteiro200 OK
PATCH✅ Sim✅ SimAtualizar campos específicos200 OK
DELETE✅ Sim❌ NãoRemover objeto204 No Content
💡 Idempotente = fazer a mesma operação várias vezes dá o mesmo resultado. Deletar o produto 1 duas vezes = mesmo efeito de uma vez. Criar com POST duas vezes = dois registros duplicados.

PUT vs PATCH — a diferença prática

PUT — substitui tudo
{
"nome": "Novo Nome",
"preco": 99.90,
"estoque": 10 ← obrigatório
}
PATCH — só o que mudou
{
"preco": 89.90 ← só isso
}

Exemplo de body no POST

# Requisição POST /api/produtos/

Códigos de status HTTP

O servidor sempre responde com um código numérico que resume o resultado. O DRF cuida disso automaticamente na maioria dos casos.

2xx — Sucesso

Código Nome Quando usar
200 OKOKGET, PUT, PATCH bem-sucedidos
201 CreatedCriadoPOST que criou um novo objeto
204 No ContentSem conteúdoDELETE bem-sucedido (sem body na resposta)

4xx — Erro do cliente

Código Nome Quando ocorre
400 Bad RequestRequisição inválidaDados inválidos no body (serializer rejeitou)
401 UnauthorizedNão autenticadoToken ausente ou inválido
403 ForbiddenSem permissãoAutenticado, mas sem permissão
404 Not FoundNão encontradoObjeto com esse ID não existe
405 Method Not AllowedMétodo não permitidoTentou DELETE numa view que não permite
422 UnprocessableNão processávelDados bem formados mas inválidos semanticamente

5xx — Erro do servidor

Código Nome Quando ocorre
500 Internal Server ErrorErro internoException não tratada no Django
503 Service UnavailableIndisponívelServidor sobrecarregado ou em manutenção
💡 No DRF, o Response(serializer.data) retorna 200, o Response(data, status=201) retorna 201. Você controla isso — mas o ModelViewSet já faz isso automaticamente.

Como uma API Django funciona — visão geral

Antes de ver cada peça em detalhe nas próximas aulas, é importante entender o fluxo completo de uma requisição no Django REST Framework.

1
urls.py — Recebe a URL, decide qual view vai lidar com ela
2
views.py — Recebe a requisição, chama o model e o serializer, retorna a response
3
serializers.py — Converte objetos Python/Django ↔ JSON, valida dados
4
models.py — Define a estrutura dos dados e se comunica com o banco

O que cada peça faz (preview)

# urls.py — mapeia URL → view
path('produtos/', ProdutoListView.as_view())

# views.py — lógica da requisição
class ProdutoListView(ListAPIView):
    queryset = Produto.objects.all()
    serializer_class = ProdutoSerializer

# serializers.py — JSON ↔ Python
class ProdutoSerializer(ModelSerializer):
    class Meta:
        model = Produto
        fields = ['id', 'nome', 'preco']
💡 Verão cada um desses arquivos em detalhes nas próximas aulas. Por agora, o importante é saber que eles existem e qual o papel de cada um.

JSON — a língua franca das APIs

APIs REST usam JSON para trocar dados. O serializer do Django é quem faz essa conversão.

# Objeto Django (Python)
produto = Produto(id=1, nome="Notebook", preco=Decimal("2999.90"))

# JSON que a API retorna
{
  "id": 1,
  "nome": "Notebook",
  "preco": "2999.90"
}

O que vem a seguir nas suas aulas

Próxima aula: URLs

Como o Django recebe uma URL e decide qual view vai processar a requisição. O DRF tem o DefaultRouter que gera automaticamente todas as URLs do CRUD.

router = DefaultRouter()
router.register(r'produtos', ProdutoViewSet)
Views

As views no DRF recebem a requisição HTTP, identificam o método (GET/POST/etc), chamam o serializer e retornam uma Response. Você vai aprender APIView, GenericAPIView e ViewSet.

Serializers

O serializer é a peça que converte seus models Django para JSON (e JSON de volta para objetos). Ele também valida os dados recebidos antes de salvar no banco.

# Validação automática
serializer = ProdutoSerializer(data=request.data)
if serializer.is_valid():
    serializer.save()  # → cria no banco
💡 Com essa base de HTTP, CRUD, REST e métodos, as views, URLs e serializers do Django vão fazer muito mais sentido. Você já sabe por que cada coisa existe.
